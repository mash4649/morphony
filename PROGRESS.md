## Phase 4: 信頼と自律性の拡大（Issue #28〜#40）

> 目標: 人間の承認頻度を段階的に下げ、自律性を安全に拡大する。
> 着手前提: MVP の運用実績（最低10タスク成功）
> 推定期間: 3〜4週間

### Issue #28: [Epic4] 信頼スコアモデルの実装
**依存**: #14（Episodic Memory）, #15（Semantic Memory）, #22（タスク完了データ保存）

**Goal**: タスク成功履歴からカテゴリ別の信頼スコアを算出し永続化する。

**Acceptance Criteria**:
- [ ] `TrustScore` モデルが定義されている（category, score: 0.0-1.0, task_count, success_count, avg_owner_rating, last_updated）
- [ ] カテゴリ別に直近20タスクの成功率 × オーナー評価平均でスコアを算出する関数がある
- [ ] オーナー評価が未入力のタスクは成功率のみで算出（評価重みは設定変更可能）
- [ ] スコアが SQLite に永続化される
- [ ] `agent trust list` でカテゴリ別信頼スコア一覧を表示できる
- [ ] 単体テストがある（スコア算出ロジックの境界値テスト）

**今回やらないこと**: 自動昇格トリガー（→ #29）、UI表示の改善

### Issue #29: [Epic4] 自律性フェーズ昇格・降格の実装
**依存**: #28（信頼スコア）, #27（autonomy_level設定）

**Goal**: 信頼スコアが閾値を超えたらオーナーに昇格提案し、品質劣化時に自動降格する。

**Acceptance Criteria**:
- [ ] 昇格条件が設定ファイルで定義できる（デフォルト: 信頼スコア 0.85 以上かつ連続5タスク成功）
- [ ] 条件達成時に `agent status` でオーナーへの昇格提案が表示される
- [ ] 昇格の最終承認は `agent approve --phase-up` でオーナーが行う（自動昇格禁止）
- [ ] 降格条件: 連続2回のオーナー評価が3以下 OR 安全性テスト違反
- [ ] 降格は自動実行され、降格イベントが監査ログに記録される
- [ ] 降格時にオーナーへ通知が送られる
- [ ] 単体テスト: 昇格条件の境界値、降格トリガーの各条件

**今回やらないこと**: Gate 1 自動承認（→ #30）、Gate 2 簡略化（→ #31）

### Issue #30: [Epic4] Gate 1 自動承認の実装
**依存**: #28（信頼スコア）, #29（フェーズ管理）, #15（Semantic Memory）

**Goal**: 過去に承認された計画と高類似度のタスクを Gate 1 で自動承認する。

**Acceptance Criteria**:
- [ ] `goal_parsed` と過去の承認済み `goal_parsed` のキーワード類似度を算出できる
- [ ] 類似度閾値（デフォルト: 0.9）と対象フェーズ（supervised 以上）が設定ファイルで変更可能
- [ ] 自動承認時は `auto_approved` フラグと類似元タスクIDが Episodic Memory に記録される
- [ ] 自動承認が発動した場合でも監査ログに記録される（透明性）
- [ ] `autonomy_level: plan_only` の場合は自動承認しない
- [ ] 自動承認後に品質劣化（オーナー評価 ≤ 2）が2回続いた場合、自動承認を一時停止する
- [ ] 単体テスト: 類似度閾値の境界値、フェーズ別の動作

**今回やらないこと**: ベクトル類似度（キーワードベースで可）、Gate 2 簡略化

### Issue #32: [Epic4] エスカレーション疲れ検知と L3→L2 降格提案

### Issue #33: [Epic4] オーナー不在パターン学習と動的タイムアウト調整
**依存**: #6（エスカレーション）, #3（イベントバス）

**Goal**: オーナーの応答速度パターンを学習し、L2/L3 タイムアウトを動的に調整する。

- [ ] 動的調整はオフにできる設定がある（`timeout_mode: static | dynamic`）

## Phase 5 → Phase 6 移行時の意思決定
```
移行前にオーナーが決定すべき事項:

1. 永続化の移行判断（必須）
   選択肢: SQLite のまま / PostgreSQL に移行
   → 目安: 同時実行エージェント数 > 3 になる場合は PostgreSQL を推奨

2. エージェント間通信方式（必須）
   選択肢: Redis Pub/Sub / RabbitMQ / gRPC / 共有DB
   → 最小コスト: 共有 PostgreSQL（追加インフラなし）
   → 推奨: Redis Pub/Sub（Phase 3 の asyncio Queue から自然に移行可能）

3. Web UI の要否
   選択肢: CLI のまま / Web UI / Grafana 等既存ツール連携
   → CLI のままでも Phase 6 は進められる

4. 第3ドメイン以降の計画
   Phase 5 の第2ドメイン追加の知見をもとに判断
```

## Phase 6 → Phase 7 移行時の意思決定
```
移行前にオーナーが決定すべき事項:

1. Best-of-N 戦略の採用（高コスト・高品質トレードオフ）
   採用する場合: 対象タスクの条件（重要度スコア閾値）を定義すること

2. メタ学習の導入判断（パターン抽出戦略の自動最適化を許可するか）
   → 暴走防止のため、最初は「1階層のみ・オーナー承認後」で開始することを推奨

3. エージェント数の目標
   → Phase 7 開始の前提は「3体以上が安定稼働していること」
```
