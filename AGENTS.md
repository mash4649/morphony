# Project Agent Rules

## Harness principle
- 状態は必ずファイルから復元する（記憶に依存しない）
- 1回のスプリントでは未完了タスク1件のみ扱う
- タスクはincrementalに進める（小さく確実に）
- タスク完了とは、実装完了・最小限の検証・progress.md更新の3点を満たすこと

## Workflow
1. progress.md を読む
2. active_task.md を読む
3. 対象タスクの完了条件を確認する
4. 必要なファイルのみ読む
5. 実装する
6. 最小限の検証を行う
7. progress.md を更新する
8. active_task.md を更新する
9. 続行条件を満たす場合のみ次のタスクに進む

## Task constraints
- 1タスク = 1機能 or 1修正
- 複数タスク同時進行は禁止
- 関係ないIssueや将来タスクには触れない
- 変更範囲は必要最小限に限定する
- 完了条件を満たさない限り「完了」としない

## Sprint contract
- 実装前に、そのタスクで何を作るか・どう確認するかを短く明示する
- 実装後は、その契約に対して合格か未達かを判定する
- 未達なら次タスクには進まず、同一タスクを継続するか停止する

## Model selection
- mini:
  - 単一ファイル
  - 定型処理
  - 既存パターン
- medium:
  - 複数ファイル（局所）
  - 通常実装
- high:
  - 横断変更
  - 難バグ
  - 設計判断
  - 仕様曖昧

## Escalation
- mini -> medium:
  - 2回失敗
  - 参照ファイルが3以上必要
  - 既存パターンで解決できない
- medium -> high:
  - 根本原因調査が必要
  - 横断変更が必要
  - 仕様解釈が必要
- high でも進行不能なら停止して報告する

## Subagents
- サブエージェントは独立タスクのみ使用
- 同じ文脈を複数エージェントに重複して渡さない
- 最大2つまで
- 親は進行管理、子は局所実装のみ担当する

## Validation
- 実装後は可能な範囲で最小限のテスト、lint、型チェック、または動作確認を行う
- 未検証部分は必ず明記する
- 評価が不合格なら次タスクへは進まない

## State management (mandatory)
- すべてのタスクは progress.md と active_task.md の更新をもって完了とする
- 更新されていない場合、そのタスクは未完了とみなす
- 実装、検証、状態更新は必ず同一実行内で行う

## Continuous execution
- 以下の条件をすべて満たす場合のみ、次の Active Task に自動で進んでよい:
  - 現在タスクが完了している
  - 検証が合格している、または未検証理由が軽微である
  - Next Queue に次タスクが存在する
  - 連続実行数が上限に達していない
  - 停止条件に該当しない

## Stop conditions
- テストまたは起動確認が失敗し、原因が特定できない
- 仕様が曖昧で判断が必要
- 想定より影響範囲が広い
- 依存追加や構成変更が必要
- 同一タスクで2回以上やり直しても前進しない
- high への昇格が必要
- セキュリティ、認証、課金、永続化破壊のリスクがある

## Required output
- 各タスク完了時に必ず以下を報告する:
  1. 今回完了したこと
  2. 検証結果
  3. progress.md更新内容
  4. active_task.md更新内容
  5. 残タスク
  6. 次の1タスク
  7. 続行したか、停止したか、その理由
