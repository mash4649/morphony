## Phase 4: 信頼と自律性の拡大

> 目標: 人間の承認頻度を段階的に下げ、エージェントの自律性を安全に拡大する。
> 理想像との対応: 「些末なことは自動実行」「人間の介入は最小限」

### 検討課題

| # | 課題 | 詳細 |
|---|------|------|
| 4-1 | 信頼度モデルの設計 | タスク成功履歴からエージェントの信頼度をどう定量化するか。カテゴリ別？全体？ |
| 4-2 | 自律性フェーズの昇格条件 | MVP→信頼獲得→成熟期への昇格をいつ・誰が・何を基準に判断するか |
| 4-3 | 昇格の安全な巻き戻し | 成熟期に昇格後に品質が劣化した場合、自動で前フェーズに戻す仕組み |
| 4-4 | Gate 1 自動承認の境界 | 類似タスクの「類似」をどう定義するか。過去5件成功？コサイン類似度0.9以上？ |
| 4-5 | Gate 2 簡略化の安全性 | 成果物確認をサマリー通知のみにした場合、品質低下を見逃すリスク |
| 4-6 | エスカレーション疲れの定量評価 | L3 頻発時の「L2 降格提案」をいつトリガーするか（直近N回で同一カテゴリがM回以上等） |
| 4-7 | オーナーの不在パターン学習 | 曜日・時間帯ごとの応答速度を学習し、タイムアウトポリシーを動的に調整 |

### 検討事項

- 信頼度は「タスクカテゴリ別」に管理すべきか、「エージェント全体」で管理すべきか
- 昇格の判断権はオーナーの明示的承認を必須とし続けるか、条件を満たせば自動化するか
- 品質劣化の検知を「オーナーフィードバック」に頼るか、「自己評価の精度」で判断するか
- 自律性拡大に伴い、監査ログの重要性がさらに増す。ログの可視化・アラートが必要

### 仮提案

- **信頼スコアモデル**: カテゴリ別に直近20タスクの成功率 × オーナー評価平均で算出
- **昇格条件**: 信頼スコア 0.85 以上かつ連続5タスク成功でオーナーに昇格提案。最終判断はオーナー
- **自動降格**: 連続2回のオーナー評価が3以下、または安全性テスト違反で即時降格
- **Gate 1 自動承認**: 過去に承認された計画と goal_parsed の類似度 0.9 以上のタスク
- **L3→L2 降格提案**: 同一カテゴリの L3 承認が連続10回すべて承認された場合に提案

## Phase 5: マルチドメイン展開

> 目標: リサーチ以外のドメイン（コード生成、データ分析、文書作成等）にも対応する。
> 理想像との対応: 「目的を与えたら、必要な手段や知識を揃え」

### 検討課題

| # | 課題 | 詳細 |
|---|------|------|
| 5-1 | ドメインの抽象化 | リサーチに特化した現在の設計を、どこまでドメイン非依存にできるか |
| 5-2 | エスカレーションルールのドメイン別管理 | コード生成では「本番デプロイ」がL3。ドメインごとにルールセットが必要 |
| 5-3 | ツールセットの爆発 | ドメインが増えるとツール数が膨大になる。選択戦略が必要 |
| 5-4 | 自己評価の汎化 | リサーチ用の4観点（達成度・信頼性・明瞭さ・効率）は他ドメインで通用しない |
| 5-5 | ドメイン間の知見転移 | リサーチで得た「情報整理」スキルはデータ分析にも転用できるか |
| 5-6 | 新ドメイン追加の手順 | オーナーが新ドメインを追加するとき、最小限何を定義すればよいか |
| 5-7 | コード実行の安全性 | コード生成ドメインでは「実行」が不可逆。サンドボックスの強化が必須 |

### 検討事項

- ドメイン定義を「ドメインプロファイル」（ツールセット + エスカレーションルール + 評価基準）として
  宣言的に管理すべきか、コードで実装すべきか
- ドメイン固有のツールはオーナーが追加するのか、エージェントが自己発見するのか
- Semantic Memory をドメイン横断で検索可能にするか、ドメイン別に分離するか
- コード実行ドメインでは Docker/Wasm ベースのサンドボックスが必要になる
- ドメイン追加のたびに計画フェーズからやり直すと重い。テンプレート化できないか

### 仮提案

- **ドメインプロファイル (YAML)**: 各ドメインを宣言的に定義
  ```
  domain:
    name: code_generation
    tools: [code_write, code_execute, test_run, lint_check]
    escalation_rules:
      - action: production_deploy → L3
      - action: test_execute → L1
    evaluation_criteria:
      - test_pass_rate: 0.0-1.0
      - code_quality_score: 0.0-1.0
      - documentation_completeness: 0.0-1.0
    sandbox: docker  # none | process | docker | wasm
  ```
- **コアエンジンのドメイン非依存化**: タスクライフサイクル・エスカレーション判定・メモリシステムは
  ドメインに依存しない。ドメイン固有部分はプロファイル＋ツール＋評価関数として注入
- **Semantic Memory のタグ付け**: パターンに `domain` タグを付けて、ドメイン内検索を優先しつつ
  `cross_domain: true` のパターンは全ドメインで参照可能にする

## Phase 6: マルチエージェント

> 目標: 複数のエージェント個体が並行して動き、知識を共有する。
> 理想像との対応: 「個体が得た経験知識スキルを他のエージェントと共有」

### 検討課題

| # | 課題 | 詳細 |
|---|------|------|
| 6-1 | エージェントのアイデンティティ | 個体を識別する仕組み。ID・名前・専門領域・信頼スコアの管理 |
| 6-2 | 知識共有プロトコル | Semantic Memory をどう共有するか。リアルタイム同期？バッチ？申請制？ |
| 6-3 | 知識の品質管理 | あるエージェントの誤った知見が他に伝播するリスク（毒知識問題） |
| 6-4 | 知識の出所追跡 | 共有された知見が「誰の・いつの・どのタスクの」経験に基づくか追跡 |
| 6-5 | 専門化 vs 汎化 | エージェントを特定ドメインに専門化させるか、全員汎用にするか |
| 6-6 | タスクの委譲・分担 | 大きなタスクを複数エージェントに分担させる仕組み |
| 6-7 | 競合・矛盾の解決 | 2体のエージェントが矛盾する知見を持っている場合の解決 |
| 6-8 | リソース競合 | 複数エージェントが同じ API やバジェットを共有する場合の制御 |
| 6-9 | オーナーの認知負荷 | エージェントが増えるほどオーナーの管理負荷が増えないか |
| 6-10 | 永続化のスケール | SQLite では複数エージェントの同時書き込みに限界がある |

### 検討事項

- 知識共有の単位は Semantic Memory のパターン単位か、Episodic Memory のエピソード単位か
- 共有された知見の信頼度をどう扱うか（受信側で再評価？送信側のスコアを引き継ぐ？）
- エージェント間の通信はメッセージキュー（Redis/RabbitMQ）か、共有データベースか
- オーナーは個々のエージェントを管理するのか、「エージェントチーム」を管理するのか
- エージェントの「死」と「生成」をどう扱うか（新しいエージェントは既存の知識を継承？）
- マルチエージェント環境でのキルスイッチの設計（個体停止？全体停止？）

### 仮提案

- **知識共有レイヤーの3層設計**:
  ```
  Layer 1: Private Memory（個体専用）
    → その個体の Episodic Memory。他のエージェントからは見えない。

  Layer 2: Shared Knowledge Base（共有知識）
    → 個体の Semantic Memory から「共有可能」と判定されたパターンを昇格。
    → 全エージェントが検索可能。ただし信頼度は受信側で再評価。

  Layer 3: Verified Knowledge（検証済み知識）
    → 複数エージェントの経験で裏付けられた高信頼パターン。
    → 新規エージェントのデフォルト知識として配布。
  ```
- **毒知識対策**: 共有パターンに `verification_count`（何体のエージェントが独立に確認したか）を
  付与。1体のみの知見は `unverified` として低優先度で提供
- **エージェントレジストリ**: 各個体の ID・専門ドメイン・信頼スコア・稼働状態を管理するメタサービス
- **永続化の移行**: SQLite → PostgreSQL。エージェント間通信は Redis Pub/Sub
- **オーナー管理UI**: CLI から Web UI へ移行。エージェント一覧・チーム管理・ダッシュボード

## Phase 7: 集合知と自己進化

> 目標: 数が増えるほど全体が賢くなり、個体の失敗が全体の資産になる。
> 理想像との対応: 「数が増えるほどに成長が進む」「１回の成功に満足せずに成果の最大化を目指す」

### 検討課題

| # | 課題 | 詳細 |
|---|------|------|
| 7-1 | 集合知の品質測定 | 「全体が賢くなっている」をどう定量化するか |
| 7-2 | 知識の陳腐化 | 時間経過で正しくなくなった知見の自動検出と更新 |
| 7-3 | メタ学習 | エージェントが「学び方を学ぶ」ことは可能か。パターン抽出戦略自体の最適化 |
| 7-4 | 自己改善の暴走防止 | 改善ループが自己強化して制御不能になるリスク |
| 7-5 | 創発的行動の検知 | 設計意図にない行動パターンが出現した場合の検知と対応 |
| 7-6 | 知識の圧縮と蒸留 | 膨大なエピソードから本質的な知見だけを抽出する高度な汎化 |
| 7-7 | エージェント間の競争と協力 | 同一タスクを複数エージェントに並行実行させ、最良結果を採用する戦略 |
| 7-8 | 新規エージェントの即戦力化 | 既存知識の一括転写で新規エージェントが即座に高い能力を持つ仕組み |
| 7-9 | 知識のバージョニング | 全体知識に「世代」を設け、巻き戻しを可能にする |
| 7-10 | 倫理・ガバナンス | 自己進化するエージェント群のガバナンスと人間の最終制御権の保持 |

### 検討事項

- 集合知の測定は「新規エージェントの初回タスク成功率」が最も直接的な指標か
- 知識の陳腐化は情報源の鮮度だけでなく、世界の変化（法改正、技術進歩等）を検知する必要
- メタ学習は計算コストが高い。どの頻度で実行するか。コスト対効果の見極めが必要
- 自己改善の暴走は「改善ループのメタ改善」が無限再帰になるリスク。再帰の深さ制限が必要
- Best-of-N 戦略（同一タスク複数実行）はコストが N 倍。コスト効率の閾値設計が必要
- 人間の最終制御権は Phase が進むほど重要度が増す。設計原則として全 Phase で維持

### 仮提案

- **集合知スコアボード**: 以下のメトリクスをダッシュボードで追跡
  - 新規エージェントの初回タスク成功率（世代を重ねるごとに向上すべき）
  - 同一カテゴリのタスク平均完了時間（経時的に短縮すべき）
  - 平均コスト効率（品質/コスト比の推移）
  - Verified Knowledge の件数と適用成功率
- **知識の鮮度管理**: Semantic Memory パターンに `last_validated` タイムスタンプを追加。
  90日以上再検証されていないパターンを自動で再検証キューに入れる
- **Best-of-N 戦略（限定的）**: 高重要度タスクのみ2体のエージェントに並行実行させ、
  結果を比較・マージ。通常タスクは1体で実行
- **メタ改善の深さ制限**: 改善ループの改善（メタ改善）は1階層まで。
  メタメタ改善は禁止（人間が判断）
- **世代管理**: 全体知識ベースに世代番号を付け、世代ごとのスナップショットを保持。
  品質劣化時に前世代にロールバック可能
