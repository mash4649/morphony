# 実装バックログ

> 凡例
> - P0: MVP リリースに必須（なければ安全に動かせない）
> - P1: MVP 品質として強く推奨（なければ実用性が低い）
> - P2: MVP 後の早期改善（運用開始後に対応可）
> - [依存] Epic間の依存を明示
> - AC = Acceptance Criteria（完了条件）

---

## Epic 1: プロジェクト基盤 & 設定管理 【P0】
> 全Epicの土台。設定スキーマ・データモデル・プロジェクト構造を確立する。
> [依存] なし（最初に着手）

### Story 1.1: プロジェクトスキャフォールド
プロジェクトのディレクトリ構成、依存管理、エントリポイントを作成する。

- **Task 1.1.1**: プロジェクトディレクトリ構成の作成
  - AC: README.md、依存管理ファイル、src/ tests/ configs/ docs/ ディレクトリが存在する
  - AC: `依存インストール → ビルド → テスト実行` のコマンドが通る

- **Task 1.1.2**: 共通型定義・データモデルの作成
  - AC: TaskState（pending/planning/approved/running/paused/
        suspended/completed/failed）の enum が定義されている
  - AC: EscalationLevel（L1/L2/L3）の enum が定義されている
  - AC: Episodic Memory スキーマ（task_id, goal, goal_parsed, plan,
        execution_log, result, self_evaluation, owner_feedback,
        extracted_lessons, metadata）が型として定義されている
  - AC: Semantic Memory スキーマ（pattern_id, category, pattern,
        confidence, source_episodes, created_at, updated_at,
        times_applied, success_rate）が型として定義されている
  - AC: Tool インターフェース（name, description, risk_level,
        cost_per_call, is_reversible, execute, validate,
        health_check）が型として定義されている

### Story 1.2: 設定管理システム
YAML ベースの設定ファイルの読み込み・バリデーション・実行時上書きを実現する。

- **Task 1.2.1**: 設定スキーマ定義とデフォルト値の作成
  - AC: agent_config.yaml のスキーマが定義されている
  - AC: escalation / budget / improvement / memory / safety の
        全セクションにデフォルト値がある
  - AC: safety.* セクションのデフォルト値は安全側に倒されている
        （sandbox_enabled: true, kill_switch_enabled: true）

- **Task 1.2.2**: 設定ローダーの実装
  - AC: YAML ファイルを読み込み、スキーマバリデーション後に設定オブジェクトを返す
  - AC: 必須フィールド欠落時に明確なエラーメッセージを出す
  - AC: 設定ファイルが存在しない場合、デフォルト設定で起動する

- **Task 1.2.3**: 実行時上書きの実装
  - AC: CLI 引数またはAPI 経由で設定値を上書きできる
  - AC: safety.* の値は CLI から緩和不可（緩和しようとした場合エラー）
  - AC: 上書きされた値がログに記録される

## Epic 3: タスクライフサイクルエンジン 【P0】
> タスクの状態遷移・チェックポイント・障害復旧を管理するコアエンジン。
> [依存] Epic 1（データモデル）, Epic 2（イベント・ログ）

### Story 3.1: タスク状態マシンの実装
タスクの状態遷移を管理する有限状態マシンを作る。

- **Task 3.1.1**: 状態遷移エンジンの実装
  - AC: TaskState 間の有効な遷移が定義されている
        （例: pending→planning, planning→approved, approved→running,
         running→paused, running→completed, running→failed,
         paused→running, any→stopped）
  - AC: 無効な遷移を試みた場合にエラーが発生する
  - AC: 状態遷移のたびに state_changed イベントが発行される
  - AC: 現在の状態と遷移履歴が永続化される

- **Task 3.1.2**: タスクキューの実装（排他制御）
  - AC: MVP では同時に running 状態のタスクは1つだけ
  - AC: 2つ目のタスク投入時は pending キューに入る
  - AC: 先行タスク完了後に自動でキューから取り出される

### Story 3.2: チェックポイント & リカバリ
ステップ単位の状態永続化と障害からの復旧を実現する。

- **Task 3.2.1**: チェックポイント保存の実装
  - AC: 各ステップ完了時にチェックポイント（完了ステップ一覧、中間成果物、
        消費バジェット）が永続化される
  - AC: チェックポイントにバージョン番号がある

- **Task 3.2.2**: 障害分類とリトライの実装
  - AC: Transient（一時的）/ Permanent（永続的）/ Fatal（致命的）の
        3分類が実装されている
  - AC: Transient → 最大3回、指数バックオフでリトライ
  - AC: Permanent → ステップスキップし代替手段を試行（ツールが別にあれば）
  - AC: Fatal → タスク停止、部分成果物を保全、オーナーに L3 報告

- **Task 3.2.3**: チェックポイントからの再開実装
  - AC: `agent resume <task_id>` で最後のチェックポイントから再開できる
  - AC: 再開時に「どこから再開するか」がログに記録される
  - AC: チェックポイントが破損している場合、エラーメッセージを出して停止する

## Epic 4: エスカレーション & 安全制御 【P0】
> L1/L2/L3 の判定・タイムアウト・キルスイッチ・バジェット制御。
> [依存] Epic 1（設定・データモデル）, Epic 2（イベント）, Epic 3（状態マシン）

### Story 4.1: エスカレーション判定エンジン
アクションごとにリスクレベルを判定し、適切な制御を行う。

- **Task 4.1.1**: リスクレベル判定ロジックの実装
  - AC: アクション名 + コンテキストから L1/L2/L3 を判定する関数がある
  - AC: リサーチドメインのアクション分類表（最小仕様 §2.2）が
        ルールとして組み込まれている
  - AC: 判定に迷った場合（ルールに該当しない場合）は上位レベルに倒す
        （Fail-safe デフォルト）

- **Task 4.1.2**: L2 通知 & タイムアウト処理の実装
  - AC: L2 アクション実行時にオーナーへ通知が送られる
  - AC: 設定値（デフォルト15分）経過後、設定されたポリシーに従って動作する
        （auto_proceed / escalate / pause）
  - AC: デフォルトポリシーは escalate（L3に昇格）

- **Task 4.1.3**: L3 承認待ち & リマインドの実装
  - AC: L3 判定でタスクが waiting_approval 状態に遷移する
  - AC: 30分経過でオーナーにリマインド通知を送る（設定変更可能）
  - AC: 24時間応答なしでタスクを自動サスペンドし、部分成果物を保全する

### Story 4.2: キルスイッチ & 安全機構
オーナーによる即時停止と異常検知を実装する。

- **Task 4.2.1**: キルスイッチの実装
  - AC: `agent stop <task_id>` で実行中のタスクが即時停止する
  - AC: 停止時に部分成果物とチェックポイントが保全される
  - AC: 停止イベントが監査ログに記録される

- **Task 4.2.2**: 異常検知の実装
  - AC: 急激なコスト増（直近5アクションのコストが平均の3倍超）を検知 → 自動停止
  - AC: 同一アクションの異常な繰り返し（同じツール+入力で5回以上）を検知 → 自動停止
  - AC: 異常検知による停止は L3 エスカレーションとしてオーナーに報告

### Story 4.3: バジェット制御エンジン
三層バジェットの消費追跡と閾値制御を実装する。

- **Task 4.3.1**: バジェットトラッカーの実装
  - AC: タスク/日次/月次の3層でコスト（金額、時間、API呼出回数）を追跡
  - AC: 各アクション実行前にバジェット残量を確認する
  - AC: 残量に応じた4段階の制御が動作する
        （>50%: L1自動, 20-50%: 効率重視モード, 5-20%: L2通知, <5%: L3停止）

- **Task 4.3.2**: バジェット消費記録の実装
  - AC: 各ツール呼び出しのコストが自動計算・記録される
  - AC: 経過時間がタスク開始からリアルタイムで追跡される
  - AC: budget_consumed イベントが発行される

## Epic 5: CLI 基盤 【P0】
> オーナーがエージェントを操作するためのコマンドラインインターフェース。
> [依存] Epic 1（設定）, Epic 3（タスク状態）, Epic 4（エスカレーション）

### Story 5.1: コア CLI コマンドの実装
タスク投入・状態確認・承認・停止の基本操作を実装する。

- **Task 5.1.1**: agent run / status の実装
  - AC: `agent run "<goal>"` でタスクが作成され task_id が返る
  - AC: `agent status` で全タスクの状態一覧が表示される
  - AC: `agent status <task_id>` で特定タスクの詳細状態が表示される
        （現在ステップ、バジェット残量、経過時間を含む）

- **Task 5.1.2**: agent approve / reject の実装
  - AC: `agent approve <task_id>` で Gate 1/L3 承認が実行される
  - AC: `agent reject <task_id> --reason "..."` で却下＋理由が記録される
  - AC: 承認/却下の対象がない場合（待機中のエスカレーションなし）にエラーメッセージ

- **Task 5.1.3**: agent stop / pause / resume の実装
  - AC: `agent stop` で即時停止（キルスイッチ）
  - AC: `agent pause` で一時停止（チェックポイント保存後に停止）
  - AC: `agent resume` でチェックポイントから再開

- **Task 5.1.4**: agent feedback の実装
  - AC: `agent feedback <task_id> --rating <1-5> --comment "..."` で
        オーナーフィードバックが Episodic Memory に記録される

### Story 5.2: 運用系 CLI コマンドの実装
ログ参照・設定管理・ヘルスチェックの CLI を実装する。

- **Task 5.2.1**: agent log / config / health の実装
  - AC: `agent log <task_id>` で監査ログを時系列表示
  - AC: `agent config show` で現在の設定を表示
  - AC: `agent config set <key> <value>` で設定変更（safety.* 緩和は拒否）
  - AC: `agent health` で各ツールの稼働状態を一覧表示

## Epic 8: 目的解釈 & 計画立案エンジン 【P1】
> オーナーの目的を解釈し、実行計画を立案する。Gate 0 → Gate 1 のフロー。
> [依存] Epic 6（ツール一覧参照）, Epic 7（Semantic Memory 参照）

### Story 8.1: 目的解釈（Goal Parser）
自然言語の目的を構造化された goal_parsed に変換する。

- **Task 8.1.1**: Goal Parser の実装
  - AC: 自然言語の goal から {対象, 調査項目, 粒度, 期限, 除外事項} を抽出する
  - AC: 抽出結果に「確信度」が付与される（LLM の推定に基づく）
  - AC: 期限が未指定の場合、デフォルト値（当日中）が適用される

- **Task 8.1.2**: 解釈検証フォーマットの実装（Gate 1 拡充）
  - AC: goal_parsed をオーナーに提示するフォーマットが定義されている
        （原文 / 解釈結果 / 確認プロンプト）
  - AC: オーナーが修正した場合、修正後の goal_parsed で上書きされる
  - AC: goal ↔ goal_parsed のペアが学習データとして蓄積される

### Story 8.2: 調査計画立案（Planner）

- **Task 8.2.1**: 計画立案エンジンの実装

  - AC: 計画の推定総コストがタスクバジェット内に収まるように調整する

- **Task 8.2.2**: Gate 1 承認フローの実装
  - AC: 計画がオーナーに提示され、承認待ち状態になる
  - AC: 承認 → running 状態に遷移
  - AC: 却下 → 却下理由がフィードバックとして記録され、再計画が可能

## Epic 9: 実行オーケストレータ 【P1】
> 計画に沿ってステップを順次実行し、各ステップでOODAループを回す。
> [依存] Epic 3（状態管理・チェックポイント）, Epic 4（エスカレーション・バジェット）,
>        Epic 6（ツール実行）, Epic 8（計画）

### Story 9.1: ステップ実行エンジン
計画のステップを順次実行する。

- **Task 9.1.1**: ステップ実行ループの実装
  - AC: 計画のステップ配列を順に実行する
  - AC: 各ステップ実行前にバジェット確認 + エスカレーション判定が行われる
  - AC: 各ステップ完了後にチェックポイントが保存される
  - AC: 各ステップの実行結果（action, input, output, duration, cost, status）
        が execution_log に記録される

- **Task 9.1.2**: ステップ内 OODA ループの実装
  - AC: 各ステップの出力を評価し、次ステップの入力を動的に調整できる

## Epic 10: 自己評価 & 改善ループ 【P1】
> 成果物の品質を自己評価し、改善サイクルを自律的に回す。
> [依存] Epic 7（メモリ記録）, Epic 9（実行・成果物生成）

### Story 10.1: 自己評価エンジン
成果物の品質を多角的に評価する。

- **Task 10.1.1**: 評価スコアリングの実装
  - AC: 4観点（目的達成度・情報信頼性・構造明瞭さ・効率）で
        各 0.0〜1.0 のスコアを算出する
  - AC: 総合スコア goal_achievement が算出される
  - AC: 各スコアに reasoning（根拠）が付与される

- **Task 10.1.2**: チェックリスト評価の実装
  - AC: 客観的チェック項目（全項目回答済み、2ソース以上、
        エビデンス付き、両論併記、鮮度明示）で合否判定される
  - AC: チェックリスト結果が self_evaluation に含まれる

- **Task 10.1.3**: ファクトチェック（簡易版）の実装
  - AC: 成果物中の事実主張に対し、別の情報源で裏取りを試行する
  - AC: 裏取り不能な主張に [要検証] ラベルが付与される
  - AC: ファクトチェックのコストがバジェットから消費される

### Story 10.2: 改善ループの実装
自己評価に基づく改善判定と反復実行を実装する。

- **Task 10.2.1**: 改善判定ロジックの実装
  - AC: goal_achievement < 0.8 で改善ループに入る（閾値は設定変更可能）
  - AC: goal_achievement ≥ 0.9 で改善ループを終了する
  - AC: 最大反復回数（デフォルト3回）に達したら終了する
  - AC: バジェット残量 < 5% で終了する

- **Task 10.2.2**: 劣化検知 & ロールバックの実装
  - AC: 改善後のスコアが前回より下がった場合、改善を中止する
  - AC: 前回の成果物（result_v{N-1}.md）を final として採用する
  - AC: 劣化検知イベントがログに記録される

- **Task 10.2.3**: 教訓抽出 & エピソード記録の実装
  - AC: タスク完了時に extracted_lessons が自動生成される
  - AC: 全データが Episodic Memory に保存される
  - AC: オーナーフィードバック（あれば）と自己評価の乖離が記録される

| **P1** | Epic 7: メモリシステム | 経験蓄積なしでは成長しない |
| **P1** | Epic 8: 目的解釈 & 計画立案 | 自律性のコア |
| **P1** | Epic 9: 実行オーケストレータ | 計画を実行に移す中核 |
| **P1** | Epic 10: 自己評価 & 改善ループ | 成果最大化の要 |
| **P1** | Epic 11: 可観測性 | 運用品質の確保 |
| **P2** | Epic 12: 成果物管理 | 初期はディレクトリ分けで代替可能 |
