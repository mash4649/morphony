### 叩き案から変更・追加した点

#### 補強① プロセスアーキテクチャの明示

叩き案に欠けている最も重要な設計判断は「CLIと実行中エージェントの通信方法」である。
`agent approve` や `agent stop` は実行中のエージェントプロセスに介入する必要がある。

**MVP方針: 共有ストア方式（SQLite + ファイルシステム）**

```
  [agent run]  ──→  Agent Process（フォアグラウンド）
                       │
                       ├─ 状態を SQLite に書き込む
                       ├─ 承認待ち時は SQLite をポーリング
                       └─ チェックポイントを JSON ファイルに保存
                       │
  [agent approve]  ──→ SQLite に承認レコードを書き込む
  [agent stop]     ──→ SQLite に停止フラグを書き込む
  [agent status]   ──→ SQLite を読み取る（読み取り専用）
  [agent log]      ──→ 監査ログファイルを読み取る（読み取り専用）
```

理由:
- HTTP サーバーやソケットを立てる必要がなく、MVP の複雑性を最小化できる
- SQLite は同一マシン上の複数プロセスからの読み書きを安全にサポートする（WALモード）
- Phase 2 で HTTP API が必要になれば FastAPI を追加するだけ。SQLite は変えない

#### 補強② 型チェッカーの追加

叩き案に型チェッカーが欠けている。Pydantic v2 で型定義しても、
呼び出し側で型チェックが通っていなければ意味がない。

**追加: `pyright`（strict モード）**

理由:
- mypy より高速で、Pydantic v2 との相性が良い（プラグイン不要で動作）
- strict モードで Issue #1 の型定義が「契約」として強制力を持つ
- CI にも組み込みやすい

#### 修正③ LLM/検索APIの位置づけを修正

叩き案は `anthropic` SDK と Tavily API を確定スタックに含めているが、
仕様上これらは Tool インターフェースの背後に隠蔽される実装詳細である。
Issue #12（MVP ツール群）着手時の選択肢として記載し、確定スタックからは外す。

理由:
- Tool インターフェース（Issue #11）がLLMプロバイダを抽象化する設計になっている
- 確定スタックに含めると、ツール差し替えの設計意図が損なわれる
- Issue #12 着手時に最新のコスト・品質を比較して選定するほうが合理的
