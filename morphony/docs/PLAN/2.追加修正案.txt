### D. 追加すべき仕様

#### D1. エラーハンドリング・障害復旧

```
障害の分類:

  一時的障害（Transient）  →  自動リトライ（最大3回、指数バックオフ）
  永続的障害（Permanent）  →  該当ステップをスキップし代替手段を試行
  致命的障害（Fatal）      →  タスク全体を安全に停止し、オーナーに報告（L3）

リカバリフロー:
  1. 各ステップ完了時にチェックポイント（状態スナップショット）を保存
  2. 障害発生時、最後のチェックポイントから再開を試行
  3. 再開不能の場合、それまでの部分成果物を保全してオーナーに報告
  4. 全障害はログに記録し、同種障害の繰り返しを Semantic Memory で学習
```

#### D2. ツール抽象化レイヤー（Plugin Architecture）

```
Tool インターフェース（概念定義）:

  Tool {
    name: string              // "web_search", "api_call", "file_read" 等
    description: string       // ツールの説明（エージェントが選択時に参照）
    risk_level: L1 | L2 | L3 // デフォルトのエスカレーションレベル
    cost_per_call: number     // 1回あたりの推定コスト
    is_reversible: boolean    // 可逆性
    execute(input) -> output  // 実行
    validate(input) -> bool   // 入力バリデーション
    health_check() -> status  // ツールの稼働状態確認
  }

MVP で実装するツール:
  - web_search    : Web検索（L1, 低コスト, 可逆）
  - web_fetch     : Webページ取得・解析（L1, 低コスト, 可逆）
  - llm_analyze   : LLMによるテキスト分析・要約（L1, 中コスト, 可逆）
  - report_render : 成果物のレンダリング（L1, 低コスト, 可逆）

拡張方法:
  - 新ツールは Tool インターフェースを実装して登録するだけで利用可能
  - risk_level / cost_per_call は管理者が上書き設定可能
  - ツール登録自体は L3 承認を必要とする（安全策）
```

#### D3. 目的解釈の検証メカニズム

```
Gate 1 の拡充:

  現状: エージェントが計画を提示 → オーナーが承認
  追加: エージェントが goal_parsed（解釈結果）も提示 → オーナーが確認

  提示フォーマット:
    ┌─────────────────────────────────┐
    │ あなたの指示:                    │
    │   "〇〇市場の競合を調べて"       │
    │                                 │
    │ 私の解釈:                       │
    │   対象: 〇〇市場                │
    │   調査項目: 主要競合プレイヤー    │
    │   粒度: 企業概要レベル           │
    │   期限: 特に指定なし → 本日中    │
    │   除外: △△は含まない（想定）     │
    │                                 │
    │ この解釈で合っていますか？        │
    └─────────────────────────────────┘

  効果:
  - 誤解釈の早期検知（手戻りコスト削減）
  - goal と goal_parsed のペアデータが蓄積され、解釈精度が向上
```

#### D4. L2 通知のタイムアウトと不在時ポリシー

```
タイムアウト設定:
  - デフォルト: 15分（オーナーが設定変更可能）
  - タイムアウト後の動作を3種から選択（設定ファイルで宣言）:

  | ポリシー | 動作 | ユースケース |
  |---------|------|-------------|
  | auto_proceed | そのまま続行 | 信頼度が高い操作向け |
  | escalate | L3に昇格（承認待ち） | 安全重視（デフォルト） |
  | pause | タスク全体を一時停止 | 長時間不在が想定される場合 |

  不在時の安全策:
  - L3承認待ちが30分を超えたらオーナーにリマインド通知
  - 24時間応答なしでタスクを自動サスペンド（部分成果物を保全）
```

#### D5. 自己評価の客観性担保

```
追加する仕組み:

  1. チェックリスト評価（構造化）
     自由記述の自己評価に加え、客観的チェック項目でスコアリング:
     - [ ] 目的に記載された全項目に回答しているか
     - [ ] 情報源が2つ以上の独立ソースに基づくか
     - [ ] 主張にエビデンス（引用・データ）が付いているか
     - [ ] 矛盾する情報がある場合、両論併記されているか
     - [ ] 情報の鮮度（直近1年以内か）が明示されているか

  2. オーナー評価との乖離追跡
     - 自己評価とオーナーのフィードバック（rating）の差を記録
     - 乖離が大きい場合、評価基準の補正に使用
     - これにより自己評価の精度が経験とともに向上

  3. 成果物のファクトチェック（簡易版）
     - 成果物中の事実主張に対し、別の情報源で裏取りを試行
     - 裏取り不能な主張には [要検証] ラベルを付与
```

#### D6. メモリ成長管理ポリシー

```
Episodic Memory:
  - 直近 100 エピソードはホットストレージ（高速検索可能）
  - 100件超過分はコールドアーカイブ（Semantic 抽出済みのもの優先）
  - アーカイブされたエピソードは要約版のみ保持（実行ログは圧縮）
  - 削除は行わない（監査要件のため）。アクセス頻度で階層化

Semantic Memory:
  - confidence が 0.3 未満かつ 90日間未使用 → 非アクティブ化（検索対象外）
  - 矛盾するパターンが検出された場合:
      → source_episodes が多い方を優先
      → 同数の場合は新しい方を優先
      → 両方の存在を記録し、オーナーに判断を仰ぐオプションも提供
  - 1カテゴリあたりのアクティブパターン上限: 50件（設定変更可能）
```

#### D7. 可観測性（Observability）

```
3つのレイヤーで可観測性を確保:

  1. イベントストリーム（リアルタイム）
     - タスク開始/完了、ステップ開始/完了、エスカレーション発生、
       エラー発生、バジェット消化をイベントとして発行
     - オーナーはサブスクライブして進捗をリアルタイム監視可能

  2. ステータスAPI（ポーリング）
     - 現在のタスク状態、完了ステップ数/全ステップ数、
       バジェット残量、経過時間を取得可能

  3. サマリーレポート（事後）
     - タスク完了時に実行サマリーを自動生成
     - 所要時間、消費コスト、エスカレーション回数、
       改善ループ回数、最終評価スコアを含む
```

#### D8. 成果物の出力形式・バージョニング

```
デフォルト出力形式: Markdown
  - 構造化されたリサーチレポート
  - セクション: 要約 / 詳細 / ソース一覧 / メタデータ
  - 必要に応じて構造化データ（JSON/YAML）を別ファイルで付与

バージョニング:
  - 改善ループの各反復で成果物をバージョン付きで保存
    例: result_v1.md, result_v2.md, result_v3.md
  - 最終版には result_final.md のエイリアスを設定
  - 劣化検知でロールバックした場合、ロールバック先が final になる

保存場所:
  - タスクIDごとのディレクトリに格納
    例: /output/{task_id}/result_v1.md
  - Episodic Memory の result フィールドからパス参照
```

#### D9. 設定管理

```
二層構造:

  1. 設定ファイル（宣言的、YAML想定）
     agent_config.yaml:
       escalation:
         l2_timeout_minutes: 15
         l2_timeout_policy: escalate    # auto_proceed | escalate | pause
         l3_reminder_minutes: 30
         l3_auto_suspend_hours: 24
       budget:
         task: { cost_usd: 5, time_minutes: 30, api_calls: 100 }
         daily: { cost_usd: 20, time_hours: 4 }
         monthly: { cost_usd: 200 }
       improvement:
         max_iterations: 3
         trigger_threshold: 0.8
         completion_threshold: 0.9
       memory:
         hot_episodes: 100
         semantic_max_per_category: 50
         inactive_threshold_days: 90
       safety:
         sandbox_enabled: true
         kill_switch_enabled: true

  2. CLI / API による実行時上書き
     - 設定ファイルをベースに、タスク投入時にパラメータを上書き可能
     - 上書きされた値は Episodic Memory の metadata に記録
     - セキュリティ関連の設定（safety.*）は CLI から緩和不可（安全策）
```

### E. 追加すべき CLI / 運用 / 検証

#### E1. CLI コマンド体系（MVP想定）

```
agent run <goal>              # タスク投入（Gate 0）
agent status [task_id]        # タスク状態確認
agent approve <task_id>       # Gate 1/L3 承認
agent reject <task_id>        # Gate 1/L3 却下（理由付き）
agent feedback <task_id>      # オーナーフィードバック入力
agent stop <task_id>          # キルスイッチ（即時停止）
agent pause <task_id>         # 一時停止
agent resume <task_id>        # 再開
agent log <task_id>           # 監査ログ表示
agent memory list             # Semantic Memory 一覧
agent memory show <pattern_id># パターン詳細
agent config show             # 現在の設定表示
agent config set <key> <val>  # 設定変更
agent tool list               # 登録済みツール一覧
agent tool add <plugin_path>  # ツール追加（L3承認が走る）
agent health                  # ヘルスチェック
```

#### E2. 運用手順

```
初回セットアップ:
  1. agent config show で初期設定を確認
  2. バジェット上限を自分の許容範囲に調整
  3. agent health で全ツールの稼働確認
  4. agent run "テスト: 日本のGDPを調べて" でスモークテスト

日常運用:
  - タスク投入 → 通知が来たら承認/却下 → 成果物受領
  - 週次で agent memory list を確認し、不要なパターンを整理

障害時:
  - agent stop <task_id> で即時停止
  - agent log <task_id> で原因調査
  - 必要に応じて agent resume <task_id> で途中再開
```

#### E3. テスト・検証戦略

```
3種類のテストを定義:

  1. シナリオテスト（E2E）
     - 事前定義されたリサーチ目的を投入し、期待される行動パターンと
       成果物の品質基準を満たすか検証
     - 例: "2024年のEV市場シェアを調べて"
       → 計画にEV関連の情報源が含まれること
       → 成果物にシェア数値とソースが含まれること

  2. リプレイテスト（回帰）
     - 過去の Episodic Memory を入力として再実行し、
       同等以上の品質スコアが出ることを確認
     - Memory やツールの変更後のデグレ検知に使用

  3. 安全性テスト
     - 意図的にスコープ外の指示を与え、L3エスカレーションが発火するか
     - 意図的にバジェット超過を誘発し、停止するか
     - 意図的に障害を注入し、リカバリが機能するか
     - 矛盾する目的を与え、確認を求めてくるか

  4. 評価ベンチマーク
     - 固定のリサーチ課題セット（10〜20題）を用意
     - 定期的に実行し、品質スコア・コスト・所要時間の推移を追跡
     - Semantic Memory の蓄積による改善度を定量測定
```

### F. brownfield 配慮（途中導入・共存策）

| # | 観点 | 方針 |
|---|------|------|
| F1 | 既存ツール・ワークフローとの共存 | ツール抽象化レイヤー（D2）により、既存の検索スクリプトやAPIラッパーを Tool として登録可能にする。既存資産の書き直しは不要 |
| F2 | 既存データの取り込み | Episodic Memory への手動インポート機能を用意。過去のリサーチ成果をエピソードとして登録し、Semantic Memory の初期学習に活用 |
| F3 | 段階的導入 | まず「計画立案＋承認」のみ（Gate 0→Gate 1）で運用開始し、自律実行は段階的に有効化できる設計にする。設定ファイルの `autonomy_level: plan_only | supervised | autonomous` で制御 |
| F4 | 設定の上書き・継承 | 設定ファイルは include / override の階層構造を持たせる。チーム共通設定 → 個人設定 → タスク固有設定 の順で上書き |
| F5 | メモリのマイグレーション | メモリスキーマにバージョン番号を付与。スキーマ変更時はマイグレーションスクリプトで既存データを変換。破壊的変更は行わず、フィールド追加のみ（append only） |
| F6 | プラグインの後付け登録 | ツール・評価関数・メモリストアは全てインターフェース経由でアクセスし、実装を差し替え可能にする。MVP実装をデフォルトとして提供しつつ、カスタム実装で上書き可能 |

### G. この修正案での非目標

| # | 非目標 | 理由 |
|---|--------|------|
| G1 | 分散システムとしてのスケーラビリティ設計 | MVP は単一プロセス前提。Phase 2 以降で検討 |
| G2 | Web UI / ダッシュボードの実装 | MVP は CLI + イベントストリームで十分。UI は Phase 2 |
| G3 | マルチテナント対応 | MVP は単一オーナー前提 |
| G4 | ツール自動発見・自動登録 | MVP ではオーナーが明示的に登録。自動発見は Phase 3 |
| G5 | 自然言語以外の入出力（音声・画像等） | テキストベースのリサーチに集中 |
| G6 | Semantic Memory の自動削除 | 安全側に倒し、非アクティブ化のみ。削除は人間判断 |

### H. 優先順位

修正案を実装優先度で分類する。

#### Tier 1: MVP リリースに必須（これがないと安全に動かせない）

| # | 項目 | 理由 |
|---|------|------|
| H1 | D1: エラーハンドリング・障害復旧 | 障害時にデータ喪失や暴走のリスク |
| H2 | D4: L2タイムアウト・不在時ポリシー | 人間不在時にエージェントが無限待機または無制御で続行 |

#### Tier 2: MVP 品質として強く推奨
