承知しました。パイロットドメインは「リサーチ」、エスカレーション基準は「リスクレベル/不可逆性」、残りはベストプラクティスに基づく推奨案で、安全面を重視した最小仕様を策定します。

```markdown
## 最小仕様（MVP）

> パイロットドメイン: リサーチタスク（情報収集・整理・要約・分析）
> スコープ: 単一オーナー／単一エージェントでの自律的リサーチ遂行

---

### 1. システム概要

人間が「リサーチ目的」を与えると、エージェントが以下を自律的に遂行する。

  目的の解釈 → 調査計画の立案 → 情報収集 → 整理・分析 → 成果物の生成 → 自己評価 → 改善

人間は「目的の入力」「重大判断の承認」「成果物の受領」のみに関与する。

---

### 2. 判断エスカレーションの閾値モデル

#### 2.1 基本原則

エージェントの全アクションを3段階に分類し、段階に応じて制御する。

| レベル | 名称 | 定義 | エージェントの振る舞い |
|--------|------|------|----------------------|
| L1 | 自動実行 | 可逆かつ低リスク | 即座に実行。事後ログのみ記録 |
| L2 | 通知付き実行 | 可逆だが中リスク、または判断に曖昧性がある | 実行しつつ人間に通知。一定時間内に人間が否認しなければ続行 |
| L3 | 承認必須 | 不可逆、高リスク、またはスコープ外 | 人間の明示的な承認を得るまで待機 |

#### 2.2 リサーチドメインでの具体的分類

| アクション | レベル | 理由 |
|-----------|--------|------|
| 公開情報の検索・閲覧 | L1 | 可逆・副作用なし |
| 調査計画の変更（小規模） | L1 | 可逆・低コスト |
| 調査計画の大幅な方針転換 | L2 | 方向性ミスのリスク |
| 調査範囲の拡大（コスト増を伴う） | L2 | コスト影響あり |
| 外部APIへの課金を伴う呼び出し | L2 | 金銭コスト |
| 最終成果物の確定・納品 | L3 | 不可逆（相手に届く） |
| 新たな外部サービスへの接続・認証 | L3 | セキュリティリスク |
| 個人情報・機密情報を含む操作 | L3 | プライバシーリスク |
| 当初の目的スコープを超える判断 | L3 | オーナーの意図から逸脱するリスク |

#### 2.3 安全設計原則

- **Fail-safe デフォルト**: 判定に迷った場合は上位レベル（より安全側）に倒す
- **エスカレーション疲れ防止**: L3が頻発する場合、パターンを学習しオーナーに
  「このカテゴリをL2に下げてよいか」提案する（メタ判断もオーナー承認制）
- **監査ログ**: 全レベルのアクションをタイムスタンプ付きで不変ログに記録
- **キルスイッチ**: オーナーはいつでもエージェントを即時停止できる

### 3. 経験蓄積の最小データモデル（推奨案）

#### 3.1 二層メモリアーキテクチャ

```
┌─────────────────────────────────────────┐
│            Semantic Memory              │
│    （汎化された知見・パターン・ルール）      │
│                                         │
│  "学術論文の調査では Google Scholar →     │
│   引用数フィルタ → アブストラクト精読 の    │
│   順序が最も効率的だった"                  │
├─────────────────────────────────────────┤
│            Episodic Memory              │
│    （個別タスクの実行記録）                 │
│                                         │
│  タスクID: R-0042                        │
│  目的: "〇〇市場の競合調査"               │
│  計画 → 実行ログ → 結果 → 自己評価        │
└─────────────────────────────────────────┘
```

#### 3.2 Episodic Memory（エピソード記憶）のスキーマ

1タスク＝1エピソードとして、以下を記録する。

| フィールド | 内容 | 例 |
|-----------|------|-----|
| task_id | 一意識別子 | R-0042 |
| goal | オーナーが与えた目的（原文） | "〇〇市場の主要プレイヤーと市場規模を調査" |
| goal_parsed | エージェントが解釈した目的 | {対象, 求める情報, 粒度, 期限} |
| plan | 立案した調査計画 | ステップ配列 |
| execution_log | 各ステップの実行記録 | [{action, input, output, duration, cost, status}] |
| result | 最終成果物 | 成果物へのリファレンス |
| self_evaluation | 自己評価 | {goal_achievement: 0.0-1.0, reasoning, improvements} |
| owner_feedback | オーナーからのフィードバック（任意） | {rating, comment} |
| extracted_lessons | このエピソードから抽出した教訓 | ["〇〇は△△より効率的", ...] |
| metadata | 実行環境・コスト等 | {total_cost, total_duration, model_used, timestamp} |

#### 3.3 Semantic Memory（意味記憶）のスキーマ

エピソードから汎化された再利用可能な知見。

| フィールド | 内容 |
|-----------|------|
| pattern_id | 一意識別子 |
| category | タスクカテゴリ（例: 市場調査, 技術調査, 文献レビュー） |
| pattern | 汎化されたルール/手順/知見（自然言語） |
| confidence | 信頼度（裏付けとなるエピソード数に基づく） |
| source_episodes | 根拠となるエピソードIDのリスト |
| created_at | 初回生成日時 |
| updated_at | 最終更新日時 |
| times_applied | 適用された回数 |
| success_rate | 適用時の成功率 |

#### 3.4 蓄積・検索の基本フロー

1. タスク完了時 → Episodic Memory に記録
2. 一定数のエピソードが溜まった時 → パターン抽出 → Semantic Memory に昇格
3. 新タスク開始時 → 目的に類似する Semantic Memory を検索 → 計画立案に反映
4. Semantic Memory の confidence / success_rate が低下 → 自動で再評価・更新

### 4. 自律改善ループの定義（推奨案）

#### 4.1 改善サイクル（OODA + Retrospective）

```
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │    ① 目的の受領・解釈                             │
  │         ↓                                        │
  │    ② 計画立案（過去の Semantic Memory 参照）        │
  │         ↓                                        │
  │    ③ 実行（各ステップで Observe → Orient →         │
  │         Decide → Act を回す）                     │
  │         ↓                                        │
  │    ④ 成果物生成                                   │
  │         ↓                                        │
  │    ⑤ 自己評価（目的達成度・品質・効率を採点）        │
  │         ↓                                        │
  │  ┌─ ⑥ 改善判定 ─────────────────────────────┐     │
  │  │                                         │     │
  │  │  改善余地あり          十分な品質         │     │
  │  │  かつコスト余裕あり     またはコスト上限    │     │
  │  │      ↓                    ↓             │     │
  │  │  ②に戻る              ⑦ 完了報告         │     │
  │  │  （改善点を反映）       ＋教訓抽出         │     │
  │  └─────────────────────────────────────────┘     │
  │                                                  │
  └──────────────────────────────────────────────────┘
```

#### 4.2 改善トリガーと終了条件

| 条件 | 判定 |
|------|------|
| 改善ループに入るトリガー | 自己評価の goal_achievement < 0.8、または明確な改善点を検出 |
| 改善ループの最大反復回数 | デフォルト3回（オーナーが設定変更可能） |
| 改善ループの終了条件 | goal_achievement ≥ 0.9、または反復上限到達、またはコスト上限到達 |
| 劣化検知 | 前回より評価が下がった場合、前回の成果物を保持し改善を中止 |

#### 4.3 自己評価の観点（リサーチドメイン）

| 観点 | 評価基準 |
|------|---------|
| 目的達成度 | 求められた情報がどの程度網羅されたか |
| 情報の信頼性 | ソースの質・多様性・鮮度 |
| 構造の明瞭さ | 成果物が読みやすく整理されているか |
| 効率 | 使用したステップ数・時間・コストは妥当か |

### 5. タスク分解の自律性レベル（推奨案）

#### 5.1 承認ゲートモデル

```
  オーナー: 「〇〇について調査して」
       ↓
  ━━━━ Gate 0: 目的入力（人間） ━━━━
       ↓
  エージェント: 目的を解釈 → 調査計画を立案
       ↓
  ━━━━ Gate 1: 計画承認（人間） ━━━━   ← 初期MVPでは必須
       ↓                                 将来的には L1/L2 タスクは自動承認
  エージェント: 計画に沿って自律実行
       │
       ├─ サブタスク分解・順序決定 → 自律（L1）
       ├─ 情報源の選定・切り替え → 自律（L1）
       ├─ 調査方針の微調整 → 自律（L1）
       ├─ 調査範囲の大幅変更 → 通知（L2）
       └─ スコープ外への逸脱 → 承認待ち（L3）
       ↓
  エージェント: 成果物ドラフト生成
       ↓
  ━━━━ Gate 2: 成果物確認（人間） ━━━━
       ↓
  オーナー: 承認 or フィードバック
       ↓
  （フィードバックがあれば改善ループへ）
```

#### 5.2 自律性の段階的拡大

| フェーズ | Gate 1（計画承認） | 実行中の自律性 | Gate 2（成果確認） |
|---------|-------------------|---------------|-------------------|
| MVP初期 | 必須 | L1のみ自動 | 必須 |
| 信頼獲得後 | 類似タスクは自動承認 | L1+L2自動 | 必須 |
| 成熟期 | 全自動（異常時のみ報告） | L1+L2自動 | サマリー通知のみ |

信頼度はタスクの成功履歴に基づいて自動計算するが、
フェーズ昇格の判断自体はオーナーの明示的承認を必要とする（安全策）。

### 6. コスト・リソース制約の運用モデル（推奨案）

#### 6.1 三層バジェットモデル

| 層 | 名称 | スコープ | 設定例 |
|----|------|---------|--------|
| 層1 | タスクバジェット | 1タスクあたりの上限 | 金額: $5 / 時間: 30分 / API呼出: 100回 |
| 層2 | 日次バジェット | 1日あたりの全タスク合計上限 | 金額: $20 / 時間: 4時間 |
| 層3 | 月次バジェット | 月間の総上限（ハードキャップ） | 金額: $200 |

#### 6.2 制御フロー

```
  各アクション実行前:
    残バジェットを確認
        ↓
    ┌─ 残 > 50% → 自動実行（L1）
    ├─ 残 20〜50% → ソフト警告をログに記録、効率重視モードに切替
    ├─ 残 5〜20% → オーナーに通知（L2）「残りわずか。継続しますか？」
    └─ 残 < 5% → 実行停止（L3）オーナーの承認なしに続行不可
```

#### 6.3 コスト最適化の自律行動

- 低コスト手段を優先する（例: キャッシュ済み情報の再利用、軽量モデルの優先利用）
- コスト効率を Episodic Memory に記録し、次回以降に反映
- 改善ループの各反復でROI（品質向上幅 / 追加コスト）を評価し、
  ROIが閾値以下なら改善を打ち切る

### 7. 安全設計（横断的要件）

#### 7.1 安全原則

| # | 原則 | 実装方針 |
|---|------|---------|
| SF1 | 最小権限の原則 | エージェントに与える権限は必要最小限。新しい権限の取得はL3承認 |
| SF2 | フェイルセーフ | 不明な状況ではアクションを止め、人間に委ねる |
| SF3 | 透明性 | 全アクション・全判断根拠をログに記録し、オーナーが事後検証可能 |
| SF4 | 可逆性の優先 | 可能な限り可逆なアクションを選択する。不可逆操作前は必ずL3承認 |
| SF5 | 劣化防止 | 改善ループで品質が下がった場合、前回の成果物にロールバック |
| SF6 | スコープ固定 | 与えられた目的の範囲外への逸脱を検知し、自動停止する |

#### 7.2 具体的な安全機構

- **キルスイッチ**: オーナーが即時に全プロセスを停止できるインターフェース
- **サンドボックス実行**: エージェントのアクションは隔離環境で実行（外部影響を制限）
- **レート制限**: API呼び出し・外部アクセスに独自のレート制限を設定
- **異常検知**: 通常パターンから逸脱した行動（急激なコスト増、異常な繰り返し等）を検知 → 自動停止
- **監査ログの不変性**: エージェント自身がログを改竄できない設計

## 次のステップ: 追加修正案に向けた確認事項

最小仕様に対して、以下の観点で追加修正案を策定する準備があります。

1. **リサーチの成果物形式** — マークダウンレポート／構造化JSON／その他、
   どの形式での出力を想定するか
2. **情報源の範囲** — Web検索のみか、特定のDB・API・社内文書も含むか
3. **オーナーとのインターフェース** — CLI／Web UI／チャット／通知手段は何を想定するか
4. **Semantic Memory の汎化タイミング** — エピソード何件から自動抽出するか、
   または手動トリガーか
5. **改善ループ中の人間フィードバック** — 改善途中でオーナーが方向修正する
   インターフェースが必要か
```
